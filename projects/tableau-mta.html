Design — Layout, Hierarchy, Story
Which metrics connect? What's seen first, second? Answer why — until you can't go deeper.
Map metrics to coordinator decisions — urgency sets the order
BANs: daily alert — act instantly. Borough panels: where to act now. OMNY bullet bar: which fare class needs assistance. KPI table: shift strategy. Map: geographic density overview.
INSTANT → LOCAL → STRATEGIC
Define visual hierarchy: 10s → 30s → 60s
10s: BANs — 3 numbers, green/red, done. 30s: Borough + OMNY — where and who. 60s: Map + KPIs — geographic drill, domain benchmarks.
STATUS → COMPARE → INVESTIGATE
Choose chart type per metric — then kill anything that doesn't justify itself
Sparklines killed: redundant with BANs. Heat map killed: Bronx/SI structurally can't hit 120K — permanent false alarm. Separate tabs killed: coordinator needs one-screen glance. OMNY BAN killed: structural metric, not daily-volatile.

Map + time block filter covers geography AND borough 120K requirement. Transfer status embedded as subtitle on Borough chart. KPIs distributed by domain, not one monolithic section.
MERGE DON'T MULTIPLY LIVE DATA · AUTO-UPDATE

Key Findings & Verdict
What the data exposed · what broke · how it was resolved
The Baseline Trap — Sunday Distortion
Anchoring to a Sunday makes weekday recovery look bigger than it is
By anchoring an index to a Sunday, a paradox emerges: modes with heavy weekday swings appear to have "recovered" more than they actually have. Monday's normal commuter spike reads as +160% growth — when it's just the weekly cycle repeating. The index is technically correct and completely misleading.
Verdict: A Cycle Plot would solve this by separating day-types. For this dashboard, the weekly average in the comparison table smooths the distortion. The paradox is real — the workaround is pragmatic.
The Aggregation Paradox — Simpson's Paradox
Aggregating all transit modes into one total hides what's actually alarming
Aggregating 7 modes into "MTA Total" produces a number that looks stable while individual modes tell completely different stories — SIR dropped 35% WoW, B&T only 8%, Subway in the middle. The total shows −26% and buries both extremes. But the plot twist: showing all 7 modes on one chart creates a 1000x scale problem (Subway at 2.1M vs SIR at 2.6K). The aggregation that hides the problem also prevents a worse visual problem.
Verdict: Small multiples for borough panels. Mode-level detail pushed to tooltips — guided by subtitle: "Hover to explore individual modes." Tooltips can be counterproductive, but with explicit instruction they're better than a chart that makes SIR invisible.
